# CnRV Mod Activity — Android + Windows

Same app as the web dashboard, now packaged as:
- **Android**: a Capacitor-wrapped native app (`android/` — a real Android Studio/Gradle project)
- **Windows**: an Electron desktop app (`electron/` — a native window, not a browser tab)

Both load the same bundled web app in `www/`, built from `src/app.jsx` with esbuild + Tailwind. Nothing here depends on a CDN or internet connection at runtime — everything needed to render the UI is bundled locally. (Google Fonts is still loaded from a CDN for nicer typography; if it's unreachable, the app falls back to system fonts and keeps working.)

## Why this needs your machine or GitHub Actions

I built and verified this project up to the point of generating a real, valid Android Gradle project and a real Electron project — but I can't compile a signed `.apk` or `.exe` inside the sandboxed environment I run in (no Android SDK, no Windows build tools, no code-signing certs, and Electron's own binary can't even be downloaded there). Two ways to get the actual installable files:

### Option A — GitHub Actions (recommended, no local setup)
1. Push this whole folder to a new GitHub repo.
2. Go to the **Actions** tab — a workflow (`.github/workflows/build-apps.yml`) runs automatically on push.
3. When it finishes (~5–10 min), open the workflow run and download:
   - `cnrv-android-debug-apk` → `app-debug.apk`, installable on any Android phone (enable "install from unknown sources" once)
   - `cnrv-windows-installer` → an `.exe` installer and a portable `.exe`, no install needed for the portable one

This costs nothing on a public repo and requires no Android Studio or Windows machine on your end.

### Option B — Build locally
Requires Node.js 20+.

```bash
npm install
npm run build          # bundles the web app into www/dist
```

**Android** (also needs Android Studio + SDK installed):
```bash
npm run android:open   # syncs and opens the project in Android Studio
```
Then in Android Studio: Build → Build Bundle(s)/APK(s) → Build APK(s).
Or from the command line: `cd android && ./gradlew assembleDebug`

**Windows** (run this ON a Windows machine, or use Option A instead):
```bash
npm run electron:dist
```
Produces an installer and a portable `.exe` in `dist-electron/`.

To just run it as a desktop window without packaging, for testing:
```bash
npm run electron:start
```

## Project layout

```
src/app.jsx          — the actual app source (React)
src/index.css        — Tailwind entry point
www/                 — built static web app (index.html + bundled JS/CSS + PWA manifest)
android/             — native Android project (generated by Capacitor)
electron/main.js     — Electron entry point (loads www/ in a native window)
.github/workflows/   — CI that builds both automatically
```

## Making changes

Edit `src/app.jsx`, then:
```bash
npm run build            # rebuild the bundle
npx cap sync android      # if you changed anything and want to test on Android
```
Electron always loads the latest `www/` at launch — no separate sync step needed.

## Data & accounts — same as the web version

- Data is stored locally on-device (`localStorage` in the WebView/window), not in the cloud. Different phones/PCs will have separate data. Use the in-app **Data & Backup** tab to export/import a full JSON snapshot between devices.
- Default admin PIN is `admin123`, set in `src/app.jsx` (search for `admin123`) — change it before handing this to your real team, then re-run `npm run build`.

## Hosting the web version on GitHub Pages

A workflow (`.github/workflows/deploy-pages.yml`) builds and deploys automatically:

1. Push this repo to GitHub.
2. Repo Settings → Pages → under "Build and deployment", set **Source: GitHub Actions**.
3. Push to `main` (or run the workflow manually from the Actions tab).
4. Your app is live at `https://<username>.github.io/<repo-name>/` in a couple of minutes.

**Before you push: read the Security notes below.** The public template ships with an *empty* roster on purpose — do not paste your real staff data into `src/app.jsx` and commit it. Import it privately instead (see below).

### Getting your real data onto the public site, without putting it in the public repo

1. Host the empty template first (steps above).
2. Open the live site, sign in as admin, go to **Data & Backup → Restore from JSON**.
3. Import your real roster/monthly data there. It's saved to that browser's `localStorage` only — never touches the GitHub repo or anyone else's browser.
4. Repeat on every device/browser that needs it (there's no shared server — see Security notes).

## Security notes — read this before hosting anywhere public

I reviewed this codebase specifically for hosting it publicly. Here's what I found and what I changed:

**Fixed:**
- **Real staff PII was baked into the JS bundle.** The original build embedded the full June 2026 roster — real names, Discord IDs, internal UIDs — directly as JavaScript, which meant anyone with the public URL could read it straight out of `dist/bundle.js`, no login required, before ever touching the sign-in screen. The public template now ships with an empty roster. Your real data is in a separate file I'm giving you (`PRIVATE-cnrv-real-data-backup.json`) — **do not commit that file to the repo.** Import it via Data & Backup after hosting.
- **The admin PIN was printed on the public login screen** ("Default PIN: admin123"). Removed. It's still a hardcoded value in the source for you to change (`ADMIN_PIN` near the top of `src/app.jsx`), but it's no longer displayed to visitors.
- **CSV formula injection.** Exported CSVs (names, notes, etc.) weren't sanitized against cells starting with `=`, `+`, `-`, or `@` — a crafted staff name or note could execute a formula/command if someone later opened the CSV in Excel. Now neutralized on export.
- **Added a Content-Security-Policy** restricting scripts/styles/fonts to known-safe sources — real hardening, unlike the item below.

**Added because you asked for it — but read this part carefully:**
You asked for something that stops people from inspecting or saving the page. I built a version of that (`AntiInspectGuard` in the source): it disables right-click, blocks F12 and the common DevTools keyboard shortcuts, and shows a warning banner if it detects DevTools is probably open. **I want to be direct about what this actually does: nothing that matters, security-wise.** Every byte of this app — the full source, the full bundle, everything — is sent to the browser regardless. Anyone can still:
- open DevTools from the browser's menu instead of a shortcut
- run `curl` or `wget` on the URL directly
- use `view-source:` or disable JavaScript before the page loads
- read the file from any browser's cache or "Save Page As," which these tricks don't block either

Treat this purely as friction against casual copy-pasting, not as a barrier that keeps anything actually secret. If there's ever data in this app you genuinely can't afford to have someone read or copy, the fix is architectural, not cosmetic: don't ship it to the client at all — that means a real backend with server-side authentication, not a static site.

**Inherent to this architecture (not fixable without a backend) — know these before you rely on this for anything sensitive:**
- **The admin PIN is not real authentication.** It's a string comparison running entirely in the visitor's browser. Anyone who reads the bundle (which, per above, cannot be prevented) has the PIN. There's no way to make a client-side-only PIN check actually secure — this applies to every "PIN gate" of this kind, not just this app.
- **Per-staff PINs are the same story**, plus they're stored in plain text in whatever JSON backup you export/import.
- **All data lives in each visitor's own browser (`localStorage`)**, unencrypted. Anyone with access to that browser/device — or just its DevTools — can read or edit it directly, including editing their own or someone else's logged scores.
- **No server means no real access control.** The "Admin" and "Staff" modes are a UI convenience, not a security boundary — nothing stops a technically-inclined staff member from switching roles by editing local browser state.

None of this means the app is unfit for its purpose — for a mod team's internal activity tracker where the main goal is convenience and the main risk is "someone edits their own score for a laugh," this is a reasonable tradeoff. It stops being fine the moment the data involved would actually hurt someone if it leaked or got tampered with. If that's ever the case, say so and I'll help you plan the move to a real backend (e.g., Firebase/Supabase with server-side rules) instead.

## Signing for a real release (not done here)

The Android build above is a **debug** APK — installable and fully functional, but not suitable for Play Store submission (needs a signed **release** build with your own keystore). The Windows build is unsigned — Windows SmartScreen may warn on first run until it builds enough reputation, or until you sign it with a code-signing certificate. Both are normal next steps once you're past internal testing, not something to worry about for team use.
